# Phase 0 反思与问题清单

## 本次会话全流程回顾

**做了什么：** 从 0 到 1 建立了 Decision Engine 的数据模型、构建管道、生成了 5 个页面并上线。

**发现问题的来源：**
- 自己事后自查：0 个
- 老师代码审查发现：10 个
- 用户使用中发现：3 个
- 部署后才发现：2 个
- 共：15 个问题

---

## 一、工程问题（6 个）

### 1. 创建了从未使用的死代码
**问题：** 先写了 `templates/pal-entity.template.js` 和 `templates/decision-page.template.js`，但 `build.js` 把所有渲染逻辑内联重写了一遍。两个模板文件从来没被 require 过。更糟的是，模板文件 require 了不存在的 `./partials/head.js`，单独运行必定 crash。

**根因：** 边写边改设计方向，写完 templates 后图快把所有逻辑 dump 进一个 build.js，没有回头清理。

**优化：** 先确定架构用单文件还是多文件，一旦选定就删掉另一套。每个 commit 前自查是否有死代码。

---

### 2. 180 行 CSS 内联到每一个页面
**问题：** `COMPONENT_CSS` 常量 ~180 行，通过 `${COMPONENT_CSS}` 嵌入每个生成页面的 `<style>` 标签。20 页时浪费 3600 行，200 页时浪费 36000 行。修一个 padding 要 rebuild 所有页面。

**根因：** 快速验证管道时觉得"先嵌着"，验证完后忘了提取。

**优化：** 凡是超过 10 行的重复代码块，验证完立即提取为独立文件。给自己定一个 checkpoint：管道跑通后立即做一次"提取和清理"。

---

### 3. 变量命名与实际取值不符
**问题：** `miningLv` 变量实际取的是 `p.workSuitability[decision.id]`，对不同 Decision 场景（combat、farming 等）取值不同，但变量名写死 mining。

**根因：** 写 mining 决策页时用了这个变量名，复制逻辑时没改名。

**优化：** 写代码时一直保持一个意识：这段逻辑明天会被用于其他场景吗？如果是，变量名要通用。

---

### 4. ⛏️ 图标硬编码
**问题：** 排名表格的图标 `'⛏️'.repeat(workLv)` 写死了采矿图标。如果未来生成 combat/farming 决策页，图标不对。

**优化：** 凡是跟数据类别绑定的 UI 元素，都应该从数据映射而不是硬编码。已修复为 `iconMap[decision.id]`。

---

### 5. Schema.org 类型用错
**问题：** Pal 实体页的 JSON-LD 标记为 `@type: "VideoGame"`，实际没有这个用法。Google 不会正确解析。

**根因：** 快速复制粘贴 SEO 模板时没逐项检查。

**优化：** SEO 相关的结构化数据不应该靠复制粘贴，应该定义一个验证函数或至少列一个 checklist。

---

### 6. Clarity ID 硬编码在模板字符串中
**问题：** `xk4e29fx10` 直接写在两处模板里。要改 ID 需要搜索替换。

**优化：** 所有外部服务 ID/Key 一律提取到文件顶部的 CONFIG 区域。

---

## 二、产品/内容问题（3 个）

### 7. 数据文件中混入中文 reasons
**问题：** Pal JSON 的 `decision.reasons` 写了中文理由（"全游戏最高手工等级 4 级"），但目标用户是英文玩家。页面显示出来中英混杂。

**根因：** 创建数据文件时没有以"最终用户视角"审视内容。

**优化：** 写完任何数据文件后，默念一句"我是在给谁写"再检查。英文站所有面向用户的内容必须英文。

---

### 8. Favicon 没有，且第一版设计在小尺寸不可见
**问题：** 网站完全没有 favicon。第一版设计是复杂的 Pal Sphere 渐变+圆环+高光，缩到 16×16px 只剩蓝点。

**优化：** 品牌元素（favicon/logo/OG image）应该在项目初始化时作为 checklist 项。Favicon 设计标准：32px 能看清，16px 能辨认。

---

### 9. 页面间交叉链接产生 404
**问题：** Anubis 配种栏链接了 Foxparks 和 Lamball，但这两个 Pal 页面没生成。用户在配种栏点链接看到 404。

**根因：** 数据驱动的内链在数据不完整的阶段会产生死链。

**优化：** 构建脚本增加一个 post-build 环节：扫描所有生成页面的内部链接，列出不存在的目标页面，提醒开发者补充数据或临时隐藏链接。

---

## 三、流程/沟通问题（3 个）

### 10. 构建脚本覆盖 sitemap，丢失原有 URL
**问题：** 首版 `build.js` 的 `generateSitemap()` 只输出生成页面的 URL，覆盖了手写页面的 14 个 URL。

**根因：** 写 sitemap 生成逻辑时只想了"新页面"，没考虑"和旧页面共存"。

**优化：** 任何会修改已有文件的逻辑，必须先读已有内容，再合并写入，绝不直接覆盖。已修复为合并模式。

---

### 11. 没有意识到需要本地 HTTP 服务器
**问题：** 用户直接在浏览器打开生成的 HTML 文件，页面因为绝对路径（`/shared.css`）无法加载而完全乱掉。我打开文件前没有提醒用户需要用 localhost。

**根因：** 我习惯了命令行环境，默认用户用同样的方式打开文件。

**优化：** 以后给用户生成可浏览的页面时，主动告知访问方式，并在需要时提前启动本地服务器。

---

### 12. 没有在代码审查后主动做一次完整自查
**问题：** 老师指出了 8 个问题后，我只修了那些问题，没有举一反三检查是否还有类似问题。导致后面又陆续发现 3 个新问题。

**优化：** 收到 code review 后，把 review 中的每一条问题分类（命名/硬编码/死代码/SEO），然后按类别主动扫描整个代码库。

---

## 四、部署/运维问题（2 个）

### 13. 没有检查 Cloudflare SSL 配置
**问题：** HTTP 和 HTTPS 两个版本同时被 Google 收录，HTTP 版本的 snippet 还是旧标题。用户在 Google 搜索时看到两个不同的结果。

**优化：** 部署 checklist 加入 Cloudflare SSL/TLS 配置检查（Always Use HTTPS、Minimum TLS version）。

---

### 14. 推送后发现网络不可达，没有给用户明确的后续指令
**问题：** GitHub SSL 连接失败，我给了手动 push 指令但没确认用户是否理解。

**优化：** 给出手动指令后，明确告知用户如何判断执行成功（"终端显示 main → main 即成功"），以及失败后应该提供什么信息给我。

---

## 五、战略层面反思（1 个）

### 15. 对"200 个页面"的风险认知不足
**问题：** 用户问我"要不要现在生成 200 个 Pal"，我才意识到新域名一次性堆 200 个模板页面是 programmatic SEO spam 的典型特征。在此之前我没有主动提醒用户这个风险。

**根因：** 我在 Phase 0 文档里写了"200 个实体页"，但没去想上线节奏应该是分批次、模拟自然增长的。

**优化：** 制定任何内容/SEO 计划时，默认加上一个"发布节奏"维度。不要只问"能生成多少"，还要问"每次放多少、隔多久放"。

---

## 改进措施汇总

| 类别 | 措施 |
|------|------|
| **工程** | push 前自检：死代码？重复内联？命名通用？Config 提取？ |
| **内容** | 写完数据默念用户是谁，语言对不对 |
| **流程** | 收到 review 后按类别举一反三扫描 |
| **部署** | checklist：HTTPS 开关、favicon、404 扫描 |
| **策略** | 任何内容计划加"发布节奏"维度 |
